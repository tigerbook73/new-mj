# 关键设计模式

> 叙事文档：讲跨玩法通用的设计模式"是什么、为什么"。具体某个玩法怎么用这些模式，见 `variants/*.md`；取舍理由见 `decisions.md`。
> 收录标准：只收已经被 ≥2 个玩法验证过的模式，未经验证的不写在这里（见 `variant-boundary.md`）。

## 1. 事件溯源 + seed 可复现

洗牌用 state 内可序列化 PRNG；任何一局 = seed + action 序列，可重放调试。事件带 seq 与可见性（public/seat），事件日志天然支持任意玩家视角与上帝视角回放。摸牌是引擎自动转移，非玩家动作。

## 2. 可见性模型

他人手牌、牌墙对客户端不存在——server 只下发 `getPlayerView(state, seat)` 的产物。TileId 与牌面同级敏感（静态映射公开）。AI 与真人同构：AI 只消费 PlayerView + getLegalActions，物理上无法作弊。

## 3. 声明窗口与时间边界

一家出牌后，仅"有合法响应"的座位进入窗口，按各玩法自己的优先级表确定性裁决（具体优先级顺序是玩法私有的，见对应 `variants/*.md`）。core 无时间概念；超时由 server 计时并代提交 pass，与主动 pass 同型。

窗口的**存在与时长**本身是全桌可观察的信息（即使某家最终 pass，"能碰而未碰"已经是读牌信息）——这是已知且接受的信息泄漏，不做混淆处理（非商用项目的取舍，见 `decisions.md` 评审点 B）。

## 4. 客户端状态与重连

命令 ack 只回执，状态一律走事件（广播含本人，幂等应用）；进新上下文（入房/重连）ack 给快照，约定细节见 `contracts/protocol-shared.md` §2。重连为快照优先——整体替换，不做增量补发（见 `contracts/session-mechanics.md` §8）。

核心不变量：**事件重建 ≡ 直接派生**——任意时刻，"初始快照 + 应用事件流"必须与"直接调用 getPlayerView"结果一致。这是断线重连（走快照）与正常游玩（走事件流）保持一致的根基，测试策略见 `testing-strategy.md`。

## 5. "外壳 + dispatch" 分层模式（反复出现的一个结构）

这个项目在两个层级上各用了一次同一个模式，值得单独命名，因为它大概率还会在第三个层级上出现：

- **单局内部**（`packages/core`）：`engine-api` 只提供四签名（`createGame`/`applyAction`/`getLegalActions`/`getPlayerView`）和事件信封骨架，不理解任何玩法；每个玩法在 `rulesets/<id>/` 下实现完整状态机，互不 import 对方的流程代码。
- **跨局编排**（房间/会话层）：session 容器只负责生命周期骨架（第几局、要不要继续），具体"下一局谁坐庄""这局怎么计分"通过 `RulesetModule` 的 dispatch 方法交给玩法自己决定，容器本身不实现任何规则。

两次都是先吃过"通用框架先行"的亏才改成这个形状（单局内部见 `decisions.md` D12；跨局庄家轮换见 D15）。如果将来在房间之上再加一层（比如匹配/排位），预期还会是同一个模式：外壳只编排，规则交给玩法/策略实现方。

## 6. 为什么这份文档只有这五条

不是设计模式只有五个，是只有五个已经被两个真实玩法（垃圾胡、血战）验证过、值得当作"公共知识"沉淀的模式。凡是只在一个玩法里出现、或者两个玩法碰巧长得一样但没被证明必须一样的东西（比如排名策略、计分累加方式），一律不进这份文档，理由和台账见 `variant-boundary.md`。
