# 阶段 4 系列：把垃圾胡打磨到完整可玩

> 过程性文档：本阶段系列的详细实施计划，由 `plan.md` 阶段区链接过来。阶段收尾时把耐久内容按 `doc-map.md` §6 吸纳到对应文档，再删除本文件。

## Context

阶段 3（web 竖切）完成后，`plan.md` 的阶段路线从"阶段 4 = 持久化"重新规划成"把 junk 一个玩法从能跑通打磨到能完整玩"，理由是：血战接入阶段 5 时应该复用阶段 4 沉淀的 AI/UI/持久化框架，不是重新设计（详见 `plan.md` 阶段 4/5 说明）。

用户定的优先级和范围：

1. **AI 优先，确保能玩**——AI 可以弱，但必须有；这是目前唯一"卡住能不能玩"的缺口（`packages/ai` 现在完全空，没有任何补位机制，一个人只能开 4 个浏览器标签自己扮演全部座位）。
2. **小特性**：明牌模式（打牌时所有牌都是明牌，包括墙牌）、replay、明牌 replay、断线托管、黑暗模式、其他界面/操作优化（纯前端，不涉及后端）。
3. **拆分原则**：一个阶段内容太多就拆成多个阶段（照 1/1.5 的先例）。

> **4.3/4.4 范围调整**：上面"其他界面/操作优化"从立项起就没有具体条目——用户原话只圈定了"纯 UI、不涉及后端"这个边界，没列清单。最初决定拆出去推迟到有具体反馈再排期；随后用户改主意，改为单独立项为 **4.4 UI/操作优化**，由用户逐项描述具体内容，原 4.4 Replay / 4.5 持久化顺延为 4.5 / 4.6。4.4 的具体条目见下方子阶段表（收尾时补齐）。
>
> **明牌模式取消，后以调试/测试专用逃生舱复活**：本节第 2 点原本包含"明牌模式"；实施时判断"提前看到未摸的墙牌"必须新增 core 能力（`getOmniscientView`），会动到 `engine-contract.md`"冻结的四签名"之外的契约面，完整实现过一版（core 两个 ruleset + protocol schema + server 部分接线，`pnpm --filter @new-mj/core verify` 全绿）后用户权衡决定不值得这个复杂度，整版 `git restore` 撤销（未提交过）。4.3 因此只剩黑暗模式一项。后续重新分析发现该判断有误——"读牌墙+所有座位手牌"没有规则语义，不需要 `RulesetModule` dispatch，跟 `lib/invariants.ts` 的 `assertContainerUniqueness` 一样一个泛型纯函数即可覆盖两个 ruleset（`decisions.md` D19）。以此落地了一个**仅调试/测试用**的轻量版：不进正式产品 UI，环境变量 `ALLOW_DEBUG_OMNISCIENT` + 调试专用协议消息 `debug:omniscientView` 门控。4.5 的"明牌 replay"可直接复用 `getOmniscientView`，届时接上 replay 重建出的历史 state 即可，不需要重新设计。

调研发现的关键事实（决定了下面的排序）：

- `RoomService.applyPlayerAction(roomId, seat, action)` 是个干净的公开方法，不需要经过 socket/gateway 就能调——AI 对接这个方法本身没有架构障碍，不需要新协议消息。
- 现在完全没有事件日志存储（`trackEventSeq` 只存了个计数器 `lastEventSeq`，不存实际事件），replay 从零开始。
- 可见性过滤（`eventsVisibleTo`）是在 gateway 层按事件标注做的，不是 core 规则的一部分——"明牌模式"可以做成 gateway 级的房间 config 开关（跳过过滤，广播给所有人），不需要改 core 规则代码。Replay 本质是"服务器重放存好的历史事件，仍然走同一套可见性过滤"，跟直播用的是同一套基础设施，不是新架构。
- `isBot` 字段在 `Player`/`RoomPlayer` 类型里已经存在（一直硬编码 `false`），只是"怎么把它设成 true、bot 座位怎么自动出牌"这两件事没做。

**关于 AI 架构边界的简化决定**：之前讨论过要不要把 AI 的合法性/算分判断做成"只吃 PlayerView"的公共契约（`decisions.md` D18 末尾提到的想法，`architecture/key-designs.md` §2 写的"AI 物理上无法作弊"）。这一轮阶段 4 的 AI **不做**那层——AI 就跑在 server 进程里，直接调 `getLegalActions(state, seat)`（完整 state，跟 server 自己一样的访问权限），"物理上无法作弊"暂时还是口头约定不是结构性保证。理由：按 D12 的教训不预先建投机基建，"AI 可以弱但必须有"这个目标不需要这层新契约就能达成；等日后要做"AI 强度分级"或者"AI 跑在独立进程/服务"这类需求时，这个简化会成为一个明确的技术债，到时候再回头做那层重构。
