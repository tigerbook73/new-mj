## 架构决策

- **D1**（技术选型已落地稳定，不再展开）：不用 Colyseus，用 NestJS + Socket.IO——Colyseus 的可变 Schema 状态同步与纯函数引擎/事件溯源冲突，其强项是高频状态同步，麻将是低频离散事件。
- **D2**（技术选型已落地稳定，不再展开）：UI 第一版 DOM + Motion，回合制牌桌无需游戏引擎渲染。
- **D3**（已并入 `architecture/system.md` §2，原文见 git 历史）
- **D4 纯函数状态机 + 事件溯源；Room 有状态对象包裹纯核**：可复现（seed+actions）、可测试（无 I/O）、序列化免费、历史免费；纯度只要求在引擎边界。
- **D5 优先级进 core，时间留 server，事件带可见性**：规则逻辑 vs 时间/I/O 的分界；超时与主动 pass 同型；server 分发无需理解规则。
- **D6**（已被 D12 取代，旧模型对现存代码不再有解释力）：core = 基建层 + RuleSet 插件层，`variantState` 命名空间等结构后来被 D12 的"engine-api 极小外壳 + 各玩法独立状态机"取代，取代过程与理由见 D12。
- **D7 非商用，不做兼容性**：可清对局数据（保用户表），事件日志格式可自由变更；协议不做版本演进（仅 PROTOCOL_VERSION 常量提示刷新）。
- **D8 配置边界**：变体之间 RuleSet（代码），变体之内地方细则 config（数据）；结构级差异不做成配置。测试以标准配置为黄金路径，fuzz 随机 config 扫组合。
- **D9**（两个玩法均已完成，不再展开）：垃圾胡为第一玩法，最简规则快速验证 core 分层；血战为第二个实现，用两个真实实现的拉扯矫正 RuleSet 抽象，促成了 D12 的接口调整。
- **D10 除 OAuth 外全走 Socket.IO**：砍掉 tRPC/HTTP 整层（查询用 ack 模式）；握手单次 JWT 鉴权绑定 socket.data.userId，不做二次临时 token（同信道同凭据的第二道门不增加攻击成本）；服务端仅 /health。
- **D11 房间对应连续 N 局，非一局即散**：房间生命周期跨多局运行，比分逐局累加，不是打完一局就散场重开。N 的具体取值、底分倍率、庄家轮换等实现细节留待阶段 2 设计，产出并入 `protocol.md` 或 `docs/rooms.md`。
- **D12 core 结构反转：engine-api 极小外壳 + rulesets/\* 各自完整状态机 + lib 无观点积木（方案 C）**：日麻确定要做，其差异（王牌区/宝牌改变牌墙结构、四风连打等中途流局出口、"无役不能和"击穿胡牌判定∥计分分离、立直后行为模式切换）是控制流级差异，不是数据/枚举级差异，方案 A/B（共用回合循环模块）无法容纳。旧模型"通用框架 + RuleSet 插件"（D6）把变化点上移到框架层——`ruleset.ts` 8 方法接口里 `getClaimOptions`/`resolveClaims`/`evaluateWin`/`settle`/`parseConfig` 五个在全仓库范围内零消费者，接口注释已自称"junk 一次性实现的形状，血战落地会强制调整"。新模型：engine-api（`createGame`/`applyAction`/`getLegalActions`/`getPlayerView` 四签名 + 事件信封 + PlayerView 骨架，四方共同依赖）不反向了解任何玩法；每个玩法在 `rulesets/<id>/` 下实现完整状态机（自有 Action/Phase/State，互不 import 对方流程代码）；`lib/` 收纯函数积木（牌/墙/PRNG/手牌分解/容器不变量）。`variantState` 字段撤销——规则状态本身就是完整状态。实施：目录/归属搬移 + import 调整为主，逻辑本体不动（阶段 1 早期窗口，血战尚未真正接入 `applyAction` 边界，避免阶段 1.5 与日麻两次流程重构）。
- **D13 apps/server 是 monorepo 里唯一的 CommonJS 包**：其余 packages/apps 均为 ESM。NestJS 官方立场是 CJS-first、不打算迁移 ESM（[nestjs/nest#13319](https://github.com/nestjs/nest/issues/13319)），跟随官方默认构建方式摩擦更小；`packages/core`/`packages/protocol` 均用 tsup 双发 CJS+ESM，让 CJS 的 server 能直接 `require()` 它们，不必让 apps/server 迁就 ESM。这条约束还在持续生效（新增 package 时要判断走哪边），不属于可精简的一次性选型。
- **D14 server 房间编排与 Socket.IO 传输层解耦**：`RoomService` 只往类型化 `EventBus` 发域事件（`room:playerJoined`/`game:snapshot` 等），不知道 socket/座位的映射关系；座位↔socket 映射由 `RoomsGateway` 自己的 `ConnectionRegistry` 维护，房间内广播直接用 Socket.IO 原生房间功能（`client.join(roomId)`），只有 `game:snapshot`/`game:event` 这两个需要按座位单播的消息才查这张表。换传输层（如未来接 Redis pub/sub 支持多进程）时 `RoomService` 不用动。鉴权同理放在 Socket.IO 握手中间件（`server.use()`）而不是 Nest 的 `CanActivate` 守卫——守卫只保护单条消息、握手阶段不经过它，未鉴权连接会白占资源；中间件能在连接建立前拒绝。
- **D15 庄家轮换公式搬入 core，`createGame` 的庄家改为外部显式必填参数**：D11 把"庄家轮换等实现细节"明确留待阶段 2 设计（未定归属）；落地后发现两处问题需一并修：① `RoomService.computeNextDealer` 与 core 内部 `createJunkGame`/`createBloodbattlePrelude` 各自用 PRNG 独立算出一个"庄家"，两者互不相通——server 广播的 `room:dealerChanged` 对实际发牌毫无影响，且这次 PRNG 抽取还顺带污染了牌墙洗牌用的同一条 PRNG 流；② 庄家判定本质是"给定上一局结果，下一局谁坐庄"，属于可能因玩法而异的规则（架构铁律 6：用户举例无连庄/连庄可抢庄/连庄下庄轮庄/连庄叠倍有上限），不应留在 server 里按 `rulesetId` 分支。修法：`createGame` 新增必填 `dealer: SeatId` 参数（第 1 局由 server 决定=房主座位，此后每局由 core 决定），`RulesetModule` 新增第五个 dispatch 方法 `computeNextDealer(finishedState, currentDealer) → SeatId`，入参直接复用已存在的、刚结束一局的不透明 `TState`，不引入新的 `SessionState` 类型或 `Room` 字段——今天两个玩法的公式都是"不看结果，顺时针轮转"（`docs/rules-junk.md` "不记连庄"、`docs/rules-bloodbattle.md` 定稿未提连庄），没有真实消费者需要跨局记忆，按 D12 的教训不预先建投机基建；未来若要支持连庄/抢庄/叠倍，届时这个函数签名是自然的扩展点。牌墙洗牌 PRNG 不再被抽庄家污染：`createWall(createPrng(seed), ...)` 直接用种子起步，既有 seed 对应的实际牌墙顺序随之改变（预期内，无 fixture 依赖具体牌序）。
- **D16 阶段 3 web 竖切先用开发态假登录，不接入真正的 Supabase OAuth**：`apps/server` 的鉴权中间件目前是明确的 MVP 占位实现（共享 HS256 密钥签发/校验 JWT，代码注释自称"not Supabase's actual JWKS/verification mechanism，Supabase wiring is phase 4"），仓库里没有任何 Supabase 项目配置（无 env 变量、无 SDK 依赖）。真接入 Supabase OAuth 需要先有可用的 Supabase 项目（URL/anon key/JWT secret）并同步升级 server 端 JWT 校验逻辑，这些都超出"先跑通登录→大厅→牌桌竖切"的阶段目标，容易把阶段 3 卡在鉴权配置上。改法：web 端一个最简登录页（输入昵称），本地用 `jose`（浏览器 Web Crypto 兼容）对 `{sub: <昵称派生的 userId>}` 做 HS256 签名，密钥与 server 端 `jwtSecret` 的开发态 fallback（`dev-only-insecure-secret`）保持一致——这只是把 server 已经存在的"开发态共享密钥"模式复用到 client 一侧，不是新架构面。特意**不**新增一个 HTTP 端点来铸造 token（会违反 D10"服务端仅暴露 /health 一个 HTTP 端点，不引入 REST 层"的铁律）。真正的 Supabase OAuth 接入留给后续（触发条件：需要真实用户身份持久化、或阶段 4 落地 Supabase Postgres 用户表时一并做，两边都要动的活儿分两次做不如合并一次）。
- **D17**（技术栈选型已落地稳定，不再逐项展开）：web 技术栈定为 Vite + React + React Router + Tailwind CSS v4 + shadcn/ui + Zustand + Vitest + Playwright（D2 只定了"DOM + Motion"大方向，没定具体工具链）。`apps/web/tsconfig.json` 覆盖 `module`/`moduleResolution` 为 `ESNext`/`bundler` 的原因（类比 D13 对 apps/server 的 ESM 偏离）见 `apps/web/AGENTS.md`。
- **D18 web 端 `game:event` 只处理"事实型"事件，不重建完整 PlayerView**：`apps/server` 的 `RoomService.applyPlayerAction` 只广播原始 `game:event`，每次动作后不会重发 `game:snapshot`（快照只在回合开始/结束时发）；而"如何从事件流正确重建 PlayerView"（`rebuildPlayerView`）是按玩法分开实现的 core 内部代码，web 依架构铁律 6 不能 import。若要在前端做到"每次动作后立刻正确刷新画面"，唯一办法是在 web 里重写一份按玩法区分的事件解释逻辑——等于把 core 的规则代码复制一份到前端，直接违反"UI 不实现规则"。折中方案：`useSessionStore` 只对"客观事实、不涉及规则判断"的事件类型（`TurnStarted`/`TileDiscarded`/`ClaimWindowOpened`/`ClaimWindowResolved`——谁的回合、谁打了什么牌、我能声明什么）做增量更新，这几个事件两个玩法（junk/bloodbattle）的 payload 字段名恰好一致（读过两边 `view.ts` 源码确认，不是巧合，是两者都基于同一套 `EVENT_TYPES` 常量）；"规则型"事件（吃/碰/杠成立、胡牌、结算）只记日志不解析，画面等下一次 `game:snapshot`（下一局开始）整体对齐。讨论中确认这不是数据泄露顾虑——可见性过滤在事件下发时已经做完，前端懂不懂规则不影响谁能收到什么数据，纯粹是避免维护两份规则代码。是否要在 `packages/ai` 立项前把"PlayerView-only 的合法性/算分引擎"做成 core 的正式公共契约（web 提示和 AI 决策共用），留待那时候再定，见 `process/plan.md` 待办。
- **D19 全明牌（调试/测试专用）：一个泛型纯函数，不新增 `RulesetModule` dispatch 方法**：阶段 4.3 曾判断"明牌模式"必须新增 core 能力、会动到 `engine-contract.md` 冻结四签名之外的契约面，实现过一版（junk/bloodbattle 各自实现 + protocol + server 接线）后因复杂度不值得整版撤销（见 `process/plan.md`"明牌模式取消"段）。重新分析发现这个判断把需求错误地当成了需要类似 D15 `computeNextDealer` 那种"每个玩法各自实现"的 dispatch 方法——但"读出牌墙 + 所有座位手牌"没有任何规则语义（不涉及胡牌判定、合法性判断等玩法差异），跟 `lib/invariants.ts` 的 `assertContainerUniqueness`/`assertTileConservation` 是同一类：用未导出改为导出的结构类型 `TileContainerState = { wall, seats }` 做泛型约束的**一个**纯函数（`lib/omniscient.ts` 的 `getOmniscientView`）即可覆盖 junk 和 bloodbattle，不需要 `RulesetModule` 新增方法，不动四签名契约面。判定标准（供未来类似需求参考）：**是否有规则语义**——`computeNextDealer` 的公式因玩法而异（连庄/抢庄等），必须 dispatch；`getOmniscientView` 只是结构化读取，玩法之间没有分歧，泛型函数天然覆盖。访问控制上，这是跟 D12 之后阶段 4.1 AI 直连完整 `state`（`nextBotAction` 绕过 `getPlayerView`）同类的接受技术债：server 侧用环境变量 `ALLOW_DEBUG_OMNISCIENT` 门控 + 复用现有房间成员校验，不进正式产品 UI，仅供调试/测试通道使用；若未来新增的玩法破坏 `{ wall, seats }` 结构假设，该泛型函数会编译失败（显式失败而非静默错误），风险与 `assertContainerUniqueness` 已承担的技术债同类。
- **D20 `rebuildPlayerView` 补成 `RulesetModule` 第三个 dispatch 方法（跟 `computeNextDealer` 同类，不是 D19 那类泛型函数）**：阶段 4.5（Replay）需要 server 按 `room.rulesetId` 通用地"从一段历史事件重建某座位的 view"，但 `rebuildPlayerView` 在 junk/bloodbattle 里只是各自 `view.ts` 的裸导出，没有像 `getPlayerView`/`computeNextDealer` 那样接入 `RulesetModule`、经 `engine.ts` 按 `rulesetId` dispatch。判定标准复用 D19 已定下的那条：**是否有规则语义**。`getOmniscientView`（D19）没有规则语义（只读 `{wall, seats}` 结构），所以是跨 ruleset 通用的一个纯函数；`rebuildPlayerView` 需要逐个解释事件 payload 的含义，这是玩法私有逻辑（junk 和 bloodbattle 的事件类型、字段完全不同），必须走 dispatch，不能像 D19 那样绕开契约。补法：`RulesetModule` 新增第三个 dispatch 方法 `rebuildPlayerView(events, seat) → TView`，`junkRuleSet`/`bloodbattleRuleSet` 挂上已有的实现（逻辑本体不变，只是接进契约对象），`engine.ts` 新增顶层 `rebuildPlayerView(rulesetId, events, seat)`（没有 `state` 可读 `rulesetId`，只能显式传参，跟其余四个从 `state.config.rulesetId` 取的写法不同）。`packages/core/src/support/registered-rulesets.ts` 顺带简化：原来单独 import 两个 ruleset 各自的 `rebuildPlayerView` 再手工拼进测试注册表，现在直接读 `junkRuleSet.rebuildPlayerView`/`bloodbattleRuleSet.rebuildPlayerView`。
- **D21 阶段 4.1 AI 直连完整 `state`，不做"PlayerView-only 合法性/算分引擎"公共契约**：D18 结尾留了一个待定问题——要不要在 `packages/ai` 立项前，把"只吃 `PlayerView` 就能判断合法性/算分"做成 core 的正式公共契约，让 web 的提示逻辑和 AI 决策共用同一层，从而让"AI 物理上无法作弊"（`architecture/key-designs.md` §2 的说法）从口头约定变成结构性保证。阶段 4.1 落地时决定**不做**：`packages/ai` 的 `chooseAction` 本身是纯函数、只吃 `getLegalActions` 算好的合法动作数组（没有作弊空间），但**调用它的地方**——`apps/server/src/rooms/room.service.ts` 的 `nextBotAction`——直接传完整 `state` 给 `GameService.getLegalActions(room.gameState, seat)`，跟 server 自己驱动真人动作用的是同一个 `state`，不经过 `getPlayerView` 过滤。理由：按 D12 的教训不预先建投机基建，"AI 可以弱但必须有"这个阶段 4.1 的目标不需要这层新契约就能达成，新契约的唯一受益者（AI 防作弊的结构性保证）在 MVP 阶段不是真实需求（AI 是自己人代码，不是玩家可控的对手）。代价：这是一笔明确记录在案的技术债，不是疏漏——"AI 物理上无法作弊"目前只是口头约定；等日后要做"AI 强度分级"（比如高难度 AI 需要看见更多信息但普通难度不该看）或者"AI 跑在独立进程/服务"（那时候 AI 就不再自动共享 server 的内存态 `state`，被迫只能拿 `PlayerView`）这类需求时，这个简化会自然暴露出来，届时再回头做 D18 留的那层重构，不需要现在预判形状。这跟 D19 的 `ALLOW_DEBUG_OMNISCIENT` 调试通道是同一类"server 侧主动放开可见性"的受控例外，但触发条件和使用者完全不同（D19 是显式开关、人工调试；这条是默认路径、AI 代码常驻）。
- **D22 阶段 5 持久化：Prisma 放进 `apps/server`（不建 `packages/db`）+ `profiles`/`room_sessions`/`game_logs` 三表无 FK + 写入 fire-and-forget + 鉴权双路径按配置分支**：参考姊妹项目 `../ai/rag-local` 的 Supabase 方案（Prisma + `supabase start` 本地 CLI）落地时定下五个关键选择。① **Prisma schema/client 放 `apps/server` 而非独立 `packages/db`**：当前架构下只有 server 碰数据库（web/mobile 全走 Socket.IO 不直连 DB），不为假想的未来消费者预建投机的包边界（D12 教训）。用 Prisma 7 的 `prisma-client-js` 生成器（不是新默认的 `prisma-client`）——后者默认输出 ESM TS 源码 + `import.meta.url`，要靠 tsx/bundler 消费；`apps/server` 是仓库唯一 CJS 包（D13），`prisma-client-js` 产出的是可直接 `require()` 的编译好的 CJS，零额外 tsc/模块配置摩擦。Prisma 7 同时把 `datasource.url` 从 schema.prisma 移除，运行时连接改用 driver adapter（`@prisma/adapter-pg` + `pg`）传给 `PrismaClient` 构造函数，Migrate 的连接单独在 `prisma.config.ts` 配置。② **`GameLog`/`RoomSession`/`Profile` 三张表互相不建外键**：`RoomSession` 行只在会话结束时写，但 `GameLog` 行在每局结束时就要写，会话中途的局必然先于 `RoomSession` 行存在——硬 FK 会拒绝这些插入。三表各自独立、都以 `roomId`/`userId` 为准归档，不做引用完整性约束，跟 D7"非商用、单向引用 userId"的精神一致。③ **`profiles` 表用应用层 upsert 而非 Postgres trigger 同步**：Supabase 官方常见模式是 `auth.users` insert 触发器自动建 `profiles` 行，但本仓库没有任何存储过程/触发器先例，且 Prisma 的 schema diffing 对手写 SQL migration 里的触发器不友好（容易被当漂移）；改成 `auth.middleware.ts` 验证真实 Supabase token 成功后，直接调用已有的 `PersistenceService.upsertProfile`，全程留在 Prisma 掌控之下。④ **写入路径 fire-and-forget，不阻塞 `RoomService` 的同步热路径**：`handleGameEnd`/session 结束时归档只 log 失败不抛出，进程崩溃在写入完成前会丢一条归档，可接受——跟"不做崩溃恢复进行中对局"（根 `AGENTS.md` 不做清单）同一挂技术债。`PersistenceService` 每个方法在 `DATABASE_URL` 未设置时短路成 no-op/null（而不是让 Prisma 真的去连一个不存在的库）——某些沙盒环境对不可达主机不会快速失败（TCP 握手挂起而非立即 `ECONNREFUSED`），曾经真的把"跑满一局"的 e2e 测试全部拖到超时。⑤ **`auth.middleware.ts` 鉴权双路径，按 `SUPABASE_URL`/`SUPABASE_SERVICE_KEY` 是否配置分支，不是新增测试模式开关**：配置存在则用 `@supabase/supabase-js` 的 `auth.getUser(token)` 验证真实 token（委托 Supabase 自己的服务器验证，不用关心项目签的是旧版共享 HS256 密钥还是新版非对称密钥）；未配置则回退 D16 的开发态 HS256 校验——现有全部 e2e 测试（沙盒/CI 默认不设 `SUPABASE_URL`）零改动继续用旧路径。`replay:get`/`debug:replayOmniscientView` 因为可能落到这条 DB - **D23 `.env` 组织方式：根目录单一位置 + `dotenv-flow` 级联，不做 symlink，`.env.production` 用 `.local` 命名约定**：阶段 5 落地本地 Supabase 时，`apps/server` 的两处手写 `dotenv.config({ path: resolve(__dirname, ...) })`（`main.ts`/`prisma.config.ts`）先后因为 `..` 层数算错读不到根目录 `.env`。趁机重新审视 monorepo env 文件组织，定下几条：① **不做 symlink 把根目录 env 文件分发进各 package**：`apps/web` 的 Vite `envDir` 和 `apps/server` 的 `main.ts`/`prisma.config.ts` 本来就直接指向仓库根目录，不需要包内物理存在一份文件，symlink 要解决的问题不存在。② **路径定位改用 `@manypkg/find-root` 的 `findRootSync(__dirname)`**（向上找 `pnpm-workspace.yaml`）取代手数 `..`，级联加载改用 `dotenv-flow`（`.env` → `.env.[NODE_ENV]` → `.env.local` → `.env.[NODE_ENV].local`，默认不覆盖已存在的 `process.env`，保证 Render 注入的真实值优先）。两个库都是 CJS 消费方需要注意的点：`@manypkg/find-root` 是 ESM-only 包，但 Node 24（本仓库 engines 要求）原生支持同步 `require(esm)`，`apps/server` 这个仓库唯一 CJS 包（D13）不受影响。③ **根 `.env` 提交进 git,只放 Supabase CLI 每次 `supabase start` 都会生成的固定 demo 值**（本地 anon/service_role JWT 是所有本地 Supabase 项目通用的公开常量，不是密钥）；此前已经填过真实本地值的文件原样改名为 `.env.development.local`（gitignored）。`supabase/.env` 同理改名为 `supabase/.env.local`（Supabase CLI 二进制内部原生识别这个文件名，`.local` 后缀与仓库其余约定一致）。④ **建 `.env.test`，但只接进 Playwright，不碰 Jest**：`.env.test`（提交进 git）把 `SUPABASE_URL`/`SUPABASE_SERVICE_KEY`/`DATABASE_URL`/`VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` 显式置空，`apps/web/playwright.config.ts` 给两个 `webServer` 子进程分别传 `NODE_ENV=test`（server 侧，`dotenv-flow` 认）和 `--mode test`（web 侧，Vite 认）触发加载；两个工具都有"`NODE_ENV`/`mode` 为 test 时跳过 `.local` 文件"的内置约定（保证测试结果不受个人本地配置影响），`.env.development.local`/`supabase/.env.local` 因此自动被绕开，不需要像最初方案那样在 `playwright.config.ts` 里手工列一份"要清空哪些变量"的名单。`apps/server` 的 Jest（单测/e2e）和 `apps/web` 的 Vitest 依然完全不加载任何 `.env` 文件——`.env.test` 只服务于 Playwright 拉起的真实 server/web 进程，`auth.middleware.spec.ts`/`persistence.service.spec.ts`/`test/replay-db-fallback.e2e-spec.ts` 硬依赖的"Jest 进程里这些变量保持 unset"这条不受影响。⑤ **不建 `.env.production`，改用 `.env.production.local` 命名约定**（只约定命名，不现在创建文件）：真实生产值不该以明文形式活在工作区（哪怕 gitignore），Render 直接注入环境变量、不读文件；`.local` 后缀天然被 `.env.*` 通配覆盖，以后要在本地留一份生产变量备份/核对副本时用这个名字即可。
